# BUG-003 - Lack of validation for maximum number of characters in first and last name fields leads to UI design violations and price hiding

| Field | Details |
|---|---|
| **Module** | UI / Gardener Profile |
| **Category** | UI |
| **Severity** | High |
| **Priority** | High |
| **Reproducibility** | 100% |

---

## Preconditions
1. User is registered in the system
2. User is in customer mode

---

## Expected Result
The name field should enforce a maximum character limit at the registration stage, for example no more than 20 characters. If a long name does exist, the UI should truncate it with "..." to ensure the card layout and price remain always visible to the user

---

## Actual Result
The long name overflows outside the card boundaries, takes up multiple lines, and completely covers the gardener's total price. The user cannot see the price. The root cause is that the name field has no maximum character limit enforced at the point of registration

---

## Impact
> This bug has a direct negative effect on the core business flow of the platform.
> When a gardener's price is hidden due to a UI overflow caused by an excessively 
> long name, customers cannot make informed decisions about which service provider 
> to choose. This can lead to user frustration, abandoned requests, and lost revenue 
> for both the platform and its gardeners. The absence of input validation at the registration stage indicates a gap in front-end data integrity controls that requires a fix at the source.
