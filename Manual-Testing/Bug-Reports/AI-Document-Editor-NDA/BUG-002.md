# BUG-002 - AI assistant displays raw HTTP 403 CSRF error message directly to the user when generating content

| Field | Details |
|---|---|
| **Module** | AI Assistant / Content Generation |
| **Category** | Functional |
| **Severity** | High |
| **Priority** | High |
| **Reproducibility** | 100% |

---

## Preconditions
1. User is authorized and logged in to the system
2. User has opened a new file

---

## Steps to Reproduce
1. Click the "+" button in the top margin of the editor
2. Enter the following prompt: "generate scenario between 2 gods from iliada"
3. Observe the system response

---

## Expected Result
Technical errors should never be visible to end users. If the AI
cannot add content directly to the editor, the system should display
a clean and friendly error message such as:
"Content could not be added to the editor automatically, please add it manually"
without exposing any technical details. CSRF and HTTP errors are
server-side issues and should never appear in the end user interface

---

## Actual Result
When the user requests content generation, the system first displays
the following raw technical error message directly in the chat panel:

`[Error] edit_file apply-edits failed: HTTP 403 from /convert/tiptap-to-markdown: CSRF: origin not allowed`

After showing the error, the system still generates the content and
displays it in the editor. The technical error appears directly before
the generated content, which is confusing and unprofessional for the user

---

## Impact
> Exposing raw HTTP 403 and CSRF error messages directly in the user 
> interface is a serious quality and security concern. From a user 
> experience perspective, technical error codes create confusion and 
> anxiety, making the platform feel unstable and unpolished. From a 
> security perspective, exposing internal API endpoint paths and server 
> error details to end users can reveal sensitive architectural information 
> about the system.
